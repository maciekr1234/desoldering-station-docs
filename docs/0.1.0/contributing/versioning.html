Versioning Guide

This document explains how to manage documentation versions and generate changelogs.

Documentation Versioningâ€‹

The documentation site supports multiple versions using Docusaurus's built-in versioning.

Current Setupâ€‹

  • "Next ðŸš§" - The current development version (from docs/ folder)
  • Versioned releases - Snapshots created when tagging releases

Creating a New Versionâ€‹

When you're ready to release a new version:

cd docs-site

# Create a versioned snapshot (e.g., for v1.0.0)
npm run docusaurus docs:version 1.0.0

This will:

  1. Copy the current docs/ folder to versioned_docs/version-1.0.0/
  2. Copy sidebars.ts to versioned_sidebars/version-1.0.0-sidebars.json
  3. Add the version to versions.json

Version Configurationâ€‹

In docusaurus.config.ts, the versioning is configured:

docs: {
  // ...
  lastVersion: 'current',
  versions: {
    current: {
      label: 'Next ðŸš§',
      path: '',
    },
  },
},

After creating versions, you can customize labels:

versions: {
  current: {
    label: 'Next ðŸš§',
    path: '',
  },
  '1.0.0': {
    label: 'v1.0.0',
    banner: 'none',  // or 'unreleased'
  },
},

Navbar Dropdownâ€‹

The version dropdown is already configured in the navbar:

{
  type: 'docsVersionDropdown',
  position: 'right',
},

Changelog Generationâ€‹

The scripts/generate_changelog.py script generates changelogs from git commits using conventional commit format.

Conventional Commitsâ€‹

For best results, use conventional commit messages:

type(scope): description

# Examples:
feat(heater): add temperature ramp rate control
fix(pump): correct motor stall detection threshold
docs(api): update PID controller documentation
refactor(gui): simplify screen navigation

Supported types:

  • feat - New features
  • fix - Bug fixes
  • docs - Documentation
  • style - Code style changes
  • refactor - Code refactoring
  • perf - Performance improvements
  • test - Tests
  • build - Build system
  • ci - CI/CD changes
  • chore - Maintenance tasks

Generate Full Changelogâ€‹

# Generate CHANGELOG.md from all tags
python scripts/generate_changelog.py --full --output CHANGELOG.md

Generate Release Notesâ€‹

# Generate changelog since last tag
python scripts/generate_changelog.py --since v1.0.0 --version v1.1.0

# Generate as blog post for Docusaurus
python scripts/generate_changelog.py \
  --format blog \
  --since v1.0.0 \
  --version v1.1.0 \
  --output docs-site/blog/2024-01-15-release-v1.1.0.md

Commit Rangeâ€‹

# Between two tags
python scripts/generate_changelog.py --since v1.0.0 --until v1.1.0

# From a specific commit to HEAD
python scripts/generate_changelog.py --since abc1234

Release Workflowâ€‹

Here's the recommended workflow for releases:

1. Prepare Releaseâ€‹

# Ensure all changes are committed
git status

# Generate changelog preview
python scripts/generate_changelog.py --since v1.0.0

# Review and finalize CHANGELOG.md
python scripts/generate_changelog.py --full --output CHANGELOG.md

2. Version Documentationâ€‹

cd docs-site

# Create versioned documentation snapshot
npm run docusaurus docs:version 1.1.0

3. Tag Releaseâ€‹

# Commit version changes
git add .
git commit -m "docs: release v1.1.0"

# Create annotated tag
git tag -a v1.1.0 -m "Release v1.1.0"

# Push with tags
git push origin master --tags

4. Create Blog Post (Optional)â€‹

python scripts/generate_changelog.py \
  --format blog \
  --since v1.0.0 \
  --version v1.1.0 \
  --output docs-site/blog/$(date +%Y-%m-%d)-release-v1.1.0.md

Tipsâ€‹

Don't Version Too Oftenâ€‹

Only create version snapshots for significant releases. Minor documentation fixes should go into the current docs and be reflected in all users' experience.

Keep Versions Manageableâ€‹

Consider removing very old versions to keep the site maintainable:

# Remove old version
rm -rf docs-site/versioned_docs/version-0.9.0
rm docs-site/versioned_sidebars/version-0.9.0-sidebars.json
# Edit versions.json to remove the entry

Banner Configurationâ€‹

For unreleased versions or deprecated versions:

versions: {
  '2.0.0': {
    banner: 'unreleased',  // Shows "unreleased" banner
  },
  '0.9.0': {
    banner: 'unmaintained',  // Shows "unmaintained" banner
  },
},
