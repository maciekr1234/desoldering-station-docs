Heater Controller

High-level temperature control for the desoldering iron heating element.

Overview‚Äã

The HeaterController manages the entire heating subsystem, including PID temperature control, state machine management, error detection, and integration with the pump and UI systems.

Features‚Äã

  • PID-based temperature control
  • Soft Landing ‚Äî reduces overshoot by limiting power near setpoint
  • Multi-state operation (Sleep, Running, Standby, Error, Tuning)
  • Automatic state transitions based on target temperature
  • Automatic standby temperature from settings
  • Configurable timeouts (heating, standby, sleep) via settings
  • Activity-based timer reset via triggerActivity()
  • Event callbacks ‚Äî register handlers for state changes, setpoint changes, temperature reached, and errors
  • Comprehensive error detection (no handle, sensor failure, overtemperature, runaway heating, heating timeout)
  • Motion-based wake detection
  • Safety fault detection with automatic shutdown

enum class HeaterError {

Tuning Mode Error Handling‚Äã

During Tuning state (used for PID calibration), most error detection is disabled to allow manual control:

  • ‚úÖ Overtemperature ‚Äî Always checked (safety critical)
  • ‚ùå NoHandle ‚Äî Skipped during tuning
  • ‚ùå SensorFailure ‚Äî Skipped during tuning
  • ‚ùå HeatingTimeout ‚Äî Skipped during tuning

Heating Timeout Behavior‚Äã

The heating timeout only triggers when the heater is actively heating up (target temperature > current temperature). This prevents false timeout errors during:

  • Cooling down (target < current)
  • Temperature maintenance (target ‚âà current)

The timer also resets when setTargetTemperature() is called with a new value

HeaterState Enum‚Äã

    Sleep,    // Heater disabled, PWM = 0

Automatic State Transitions‚Äã

setTargetTemperature() Behavior‚Äã

When setTargetTemperature() is called:

  • If target ‚â• CONFIG_MIN_SET_TEMP and not in Error/Tuning ‚Üí transitions to Running
  • If target < CONFIG_MIN_SET_TEMP and in Running ‚Üí transitions to Sleep
// This automatically starts heating
heater.setTargetTemperature(350);  // ‚Üí Running state

// This automatically stops heating
heater.setTargetTemperature(0);    // ‚Üí Sleep state

triggerActivity() Behavior‚Äã

When triggerActivity() is called:

  • Resets the inactivity timer used for standby/sleep transitions
  • If in Sleep or Standby state with valid target temperature ‚Üí transitions to Running
  • Called automatically by pump trigger, encoder rotation, and other user interactions
// Wake from sleep/standby and reset timers
heater.triggerActivity();

Standby State Behavior‚Äã

When entering Standby state:

  • Automatically sets target to gSettings.heater.standby_temperature
  • Maintains temperature using PID control
  • Transitions back to Running when setTargetTemperature() is called

Soft Landing‚Äã

Soft Landing is a power-limiting feature that reduces maximum heater output as the temperature approaches the setpoint. This significantly reduces overshoot and provides smoother temperature control.

How It Works‚Äã

When enabled, the controller limits the maximum power output based on distance from the setpoint:

Pmax=Pmin+(100‚àíPmin)√óerrorthresholdP_{max} = P_{min} + (100 - P_{min}) \times \frac{error}{threshold}Pmax‚Äã=Pmin‚Äã+(100‚àíPmin‚Äã)√óthresholderror‚Äã

For example, with default settings (threshold=30¬∞C, min_power=40%):

  • At 30¬∞C below setpoint: max power = 100%
  • At 15¬∞C below setpoint: max power = 70%
  • At setpoint: max power = 40%

Configuration‚Äã

Soft landing settings are stored in heater_settings_t:

struct {
    bool    enabled;       // Enable/disable soft landing
    uint8_t threshold;     // Distance (¬∞C) where limiting begins
    uint8_t min_power_pct; // Minimum power cap (%) at setpoint
} soft_landing;

Example Usage‚Äã

Basic Usage‚Äã

#include "heater.h"
#include "pump.h"

HeaterController heater(CONFIG_HEATER_PWM_PIN);
PumpController pump(/* config */);

void setup() {
    // Link heater and pump
    heater.attachPump(pump);
    
    // Configure PID
    heater.setPID(1.0f, 0.8f, 3.5f, 5.0f, 0.0f);
    
    // Initialize
    if (!heater.begin()) {
        PICO_LOGE("MAIN", "Heater init failed");
        return;
    }
    
    // Set target - automatically starts heating
    heater.setTargetTemperature(350);
}

Error Handling‚Äã

void checkHeaterStatus() {
    if (heater.getState() == HeaterState::Error) {
        HeaterError err = heater.getError();
        
        switch (err) {
            case HeaterError::NoHandle:
                PICO_LOGE("UI", "Handle not connected!");
                break;
            case HeaterError::Overtemperature:
                PICO_LOGE("UI", "Overtemperature detected!");
                break;
            case HeaterError::HeatingTimeout:
                PICO_LOGW("UI", "Heating timeout - check element");
                break;
            // ... handle other errors
        }
        
        // Clear error and return to Off state
        heater.clearError();
    }
}

Using Callbacks‚Äã

The HeaterController provides a callback system for reacting to events without polling. Multiple callbacks can be registered for each event type, and they are invoked in a thread-safe manner without blocking the heater control loop.

// Callback function for state changes
void on_state_change(HeaterState old_state, HeaterState new_state, void* ctx) {
    PICO_LOGI("UI", "Heater state: %d -> %d", (int)old_state, (int)new_state);
    // Update UI, trigger sounds, etc.
}

// Callback for when target temperature is reached
void on_temp_reached(uint16_t temp, void* ctx) {
    PICO_LOGI("UI", "Target temperature %d¬∞C reached!", temp);
    // Could trigger a beep or notification
}

// Callback for errors
void on_error(HeaterError error, void* ctx) {
    PICO_LOGE("UI", "Heater error: %d", (int)error);
    // Show error dialog, play alarm sound, etc.
}

void setup() {
    // Register callbacks (returns handle for later removal)
    heater_cb_handle_t state_handle = gHeater.onStateChanged(on_state_change, nullptr);
    heater_cb_handle_t reached_handle = gHeater.onSetpointReached(on_temp_reached, nullptr);
    heater_cb_handle_t error_handle = gHeater.onError(on_error, nullptr);
    
    // ... later, to remove a callback:
    gHeater.removeCallback(state_handle);
}

Callback Events:

EventCallback TypeTriggered When
State Changedheater_state_cb_tHeater transitions between states
Setpoint Changedheater_setpoint_cb_tTarget temperature is modified
Setpoint Reachedheater_reached_cb_tTemperature first reaches target (¬±2¬∞C)
Errorheater_error_cb_tHeater enters Error state
Thread Safety

Callbacks are invoked from the heater control task but are designed to not block the control loop. A snapshot of registered callbacks is taken under mutex protection, then callbacks are invoked outside the mutex. Maximum 8 callbacks per event type are supported.

Safety Features‚Äã

Safe Mode‚Äã

When entering Error state:

  1. PWM immediately set to 0
  2. PID controller reset
  3. Error code stored
  4. Error logged via PICO_LOGE
  5. Manual clearError() required to recover

Startup Grace Period‚Äã

To prevent false runaway detection during boot:

  • Runaway detection is disabled for first 3 seconds after boot
  • Allows ADC readings to stabilize
